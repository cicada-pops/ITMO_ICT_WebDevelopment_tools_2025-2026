# Сводное сравнение и выводы

## Сводная таблица

| Задача                          | threading | multiprocessing | asyncio |
|---------------------------------|----------:|----------------:|--------:|
| Задача 1, CPU-bound (N = 10⁸)   | 2.006 с   | **0.547 с**     | 1.850 с |
| Задача 2, I/O-bound (12 URL)    | 0.706 с   | 1.190 с         | **0.741 с** |

Жирным выделены лучшие времена для каждой задачи.

## Когда какой инструмент

| Подход          | Сильная сторона                | Слабая сторона                      | Когда брать |
|-----------------|--------------------------------|-------------------------------------|-------------|
| `threading`     | Прост, дешёвый запуск          | GIL — нет параллелизма по CPU       | Лёгкий I/O, мало задач |
| `multiprocessing` | Реальный параллелизм по ядрам | Дорогой запуск, pickle, IPC        | Тяжёлая арифметика, кодеки |
| `asyncio`       | Тысячи задач, дешёвое переключение | Не помогает CPU-bound; нужен async-стек | Сеть, БД, веб-серверы |

## Ключевые наблюдения

1. **GIL — главная причина существования всех трёх API сразу**:
   потокам он мешает использовать несколько ядер, поэтому для CPU-bound
   нужен `multiprocessing`. Для I/O-bound GIL отпускается на системных
   вызовах, поэтому достаточно `threading` (или `asyncio`).
2. **Не любой параллелизм одинаково полезен**: `multiprocessing`
   ускорил арифметику в 3.7×, но *замедлил* парсинг в 1.7× — стоимость
   запуска процессов оказалась выше выгоды.
3. **`asyncio` не магия**: на CPU-bound задаче он работает практически
   как однопоток. Чтобы async-программа реально была параллельной,
   внутри корутины должна быть точка `await` на действительно
   асинхронный примитив (`aiohttp`, `asyncpg`, и т.п.).

## Возвращаясь к `N = 10^13`

Если экстраполировать результаты задачи 1 на исходное `N = 10^13`,
получится:

| Подход            | Оценка времени для N = 10¹³ |
|-------------------|-----------------------------:|
| `threading`       | ≈ 56 ч                       |
| `multiprocessing` | ≈ 15 ч                       |
| `asyncio`         | ≈ 51 ч                       |

То есть даже в самом быстром варианте задача занимала бы более полусуток
— на чистом Python такие диапазоны имеет смысл считать формулой
`N (N + 1) / 2` за O(1), а параллелизм демонстрировать на меньшем `N`,
что и сделано в этой работе.
